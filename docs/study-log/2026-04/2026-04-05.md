## 用户原始反馈

- `我觉得 langgraph 就是一个 agent 框架，可以自己定义图结构，以达到 agent 状态循环的目的，通过点和边显示定义`
- `langgraph 是一个基于图的有状态流程编排框架，因为他是图结构，不像 langchain 一样是线性结构，可以自循环检测`
- `状态 state 主要就是保存每个节点的信息`
- `如果没有 state，langgraph 会缺少信息存储能力，那样每个节点就不好交互了`
- `get_embedding 是将 str 等转化为向量`
- `retrieve_node 是讲用户的问题转化为向量并和数据库中的向量做匹配`
- `llm 是将匹配回来的知识与用户的问题都当做 system 提示词上传到 llm 组织回答`
- `rag 是基于现有的知识库让大模型回答，llm 本身没有变化`
- `微调是 llm 自身的变化`
- `embedding 在 RAG 里负责知识存储、知识检索和匹配`
- `1.rag 首先是将知识库 embeding 向量化存储，然后将用户问题 embeding，进行向量检索匹配，然后召回相关知识，返回给 llm 进行组织回答`
- `2.用户问题也需要向量化才能和知识库的进行检索匹配`
- `retrieve：主要是作检索`
- `recall：召回，将匹配的数据返回`
- `rerank：返回的数据很多，需要进行重排序匹配相似度最高的`
- `1.有可能检索回来的有很多条数据，不可能取全部`
- `2.可能其中很多数据的相关性并不高，没什么用，所以要取相似度高的`
- `RAG 的完整链路：首先将已有的知识库 embeding 转化成向量存贮在向量库中，然后用户问问题，将问题也 embeding，retrieve 检索匹配，再 recall 召回，再 rerank 对召回的数据重排序，找出相似度最高的再和问题一起传给 llm 进行组织回答`
- `降低上下文窗口、降低噪音、为 llm 补充外部知识、降低成本`

- `状态` 在图中的具体作用还没有展开到“状态如何在节点之间传递和更新”。
